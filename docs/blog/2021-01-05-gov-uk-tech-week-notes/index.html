Accessibility (Ian)

Short one - nothing to report in terms of stuff done, since the team took advantage of the festive bank holidays to take a break. In the next week we’re going to be taking a step back to see what we want and can achieve over the coming months, looking at implementing a more accessible accordion, and generally continuing the good work in making GOV.UK more accessible.

Accounts (Michael)

In the past couple of weeks we've improved our logging, recording more security events, and we've got a Splunk dashboard set up to help us spot weird trends; we've implemented the NCSC password denylist, so users can't sign up with insecure passwords; we've made our registration and sign in forms more password-manager friendly, by changing how we mark up the input elements; and finally some content updates for brexit as well.

Data Labs (Karl/Kelvin)

We gathered in our first week to plan out epics for the engineers on the team to focus on next. The first thing we’re working on is to set up another instance of neo4j to provide a users with a separate Knowledge Graph with the data they expect, but maintain a sandbox for the data scientists to continue their experiments. This “new” Knowledge Graph will be a stop-gap while we develop the Templated Queries Prototype, with the plan to re-use existing data sources that are in production when we move away from the prototype. We’re also gearing up to plan how we can extract data models from the network database that is Knowledge Graph, and represent it still as a graph in somewhere we can use in production.

DGU (Ken)

Platform Health (Tom)

Last week, we made live the new “Protected food & drink names” finder which involved running a bulk import and publish task.

We’ve also been experimenting with grouping together PagerDuty alerts to reduce the number of PagerDuty incidents we get when lots of machines have the same error.

Following on from an investigation before the new year, we’re going to be thinking about whether we should have two Sidekiq processes for Email Alert API (one for “fast jobs” such as sending emails, and one for the longer running ones) to avoid them being blocked on each other.

This week we made an improvement to Link Checker API to be more flexible with URLs which are not technically valid, but would be accepted by all known browsers. This should reduce the number of false errors and warnings in Local Links Manager.

We’ve also been working to fix a cross-site scripting vulnerability in govuk_publishing_components that was found as part of a recent round of IT health checks (pen testing).

As part of work to reduce errors in Sentry we’ve spotted a few fixes to the Publishing API to reduce the number of unactionable downstream worker errors we see.

We picked up the work importing Mapit again after Christmas and it appears to be going more smoothly this time, we should be in a position to deploy to production shortly.

Finally, we’ve been thinking about our missions in the future, so we’re doing a team exercise to identify the biggest problems on the platform and whether we could form missions out of them.

Replatforming (Chris Banks)

Bill has been adding some tests for our Terraform code and wrote a Concourse resource for GitHub commits, which gives us more flexibility about when we run CI actions.

Our config management code is about to get much simpler and more maintainable, thanks to Richard's refactoring.
